# Track 05: Messages Table Extraction

> **Status**: Spec — ready for implementation
> **Last Updated**: 2026-03-23
> **Author**: Claude (Claudius)
> **Motivation**: CLI `get` crashes on large chats; systemic SW memory pressure

---

## 1. Problem Statement

Every `Chat` record in IndexedDB is stored as a **monolithic JSON blob** — metadata,
sync state, and the entire `messages` dictionary all in one row. For most chats this
is fine. For large chats (200+ messages, embedded tool output, long code blocks) a
single row can be 10–50 MB of JSON. Loading it costs:

| Stage | What happens | Memory peak |
|---|---|---|
| `getDB().chats.get(id)` | Dexie deserialises the full row | ~1× JSON size |
| `JSON.stringify(chat)` | Serialise for WS / export | ~2× JSON size |
| WS `.send()` | Copy into Chrome message buffer | ~3× JSON size |

A 20 MB chat touches ~60 MB during a single `get`. The MV3 service worker has no
fixed memory limit, but Chrome will terminate it when the browser is under pressure,
and the JS heap limit for background pages in practice is roughly 50–100 MB. This
explains the intermittent crashes users see **even without the CLI** — the Viewer,
bulk export, and the full-text indexer all call `getChat()` and hit the same wall.

The CLI daemon makes the problem deterministic: a `haevn get` on any chat above
~150 messages reliably crashes the SW.

---

## 2. Root Cause

The `messages` dict is stored **inside** the `chats` row. IndexedDB has no concept of
field-level partial reads; `table.get(id)` always deserialises the entire record.
There is no way to load just the message IDs, or just one branch, without loading
every message first.

This is not a query optimisation problem — it is a schema problem.

---

## 3. Proposed Solution: `chatMessages` table (DB v17)

Extract `Chat.messages` into a dedicated `chatMessages` table with one row per
message. The `chats` row retains all metadata and sync fields, but stores
`messages: {}` (empty) in the DB. The populated `messages` dict that application
code expects is assembled on demand by the repository layer.

```
BEFORE (v16)
───────────────────────────────────────────────────────────────────────
chats row  { id, source, title, currentId, messages: { id0: {...},
             id1: {...}, id2: {...}, ..., idN: {...} }, lastSynced, … }

AFTER (v17)
───────────────────────────────────────────────────────────────────────
chats row  { id, source, title, currentId, messages: {}, lastSynced, … }

chatMessages rows
  { pk: "chatId:msgId", chatId, id, parentId, childrenIds, message, … }
  { pk: "chatId:msgId", chatId, id, parentId, childrenIds, message, … }
  …
```

### Why a compound primary key `[chatId+id]`?

Message IDs come from 10+ platforms. While almost all are UUIDs, some platforms
reuse short numeric IDs or opaque strings across chats. A compound key
`[chatId+id]` guarantees uniqueness regardless of platform ID semantics.

## 4. New Schema (DB Version 17)

```typescript
// db.ts — new table declaration
chatMessages!: Table<StoredChatMessage, [string, string]>;  // PK: [chatId, id]

// Version 17: extract messages into a separate table
this.version(17).stores({
  chats: "id, source, title, lastSyncedTimestamp, providerLastModifiedTimestamp, \

          [source+providerLastModifiedTimestamp], [source+title], \

          [deleted+providerLastModifiedTimestamp], [deleted+title], \
          [deleted+source+lastSyncedTimestamp], [deleted+source+providerLastModifiedTimestamp], \

  lunrIndex: "id",
  openwebuiInstances: "id, baseUrl",
  cache: "id",
  thumbnails: "++id, chatId, messageId, [chatId+messageId], source, role, mediaType, \
               generatedAt, [source+generatedAt], [role+generatedAt], \
               [mediaType+generatedAt], [source+role+generatedAt]",
  mediaContent: "id",
});
// Note: NO upgrade() handler here — migration runs as a background task (§6).
```

### New type

```typescript
// db.ts
/**
 * A ChatMessage as stored in the chatMessages table.
 * Identical to ChatMessage in the model; the chatId field acts as the
 * foreign key and is part of the compound primary key.
 */
export type StoredChatMessage = ChatMessage;  // chatId is already on ChatMessage
```

## 5. Data Model Change

`Chat.messages` becomes **optional** in the TypeScript type, but the stored DB row
always has `messages: {}`. This creates a clean API contract:

```typescript
// haevn_model.ts (modified)
export interface Chat {
  // …existing fields…
  messages: Record<string, ChatMessage>;  // {} in DB; populated by repository on demand
}
```

No type change is strictly required (it remains `Record<string, ChatMessage>`), but
callers must not assume the dict is populated after a bare `getChat()` call. The
repository layer enforces this via separate API methods (§7).

---

## 6. Migration Strategy: Background Lazy Migration

A v17 upgrade handler that iterates 2451 full chat rows in a single IndexedDB
transaction will hit the transaction timeout (~60 s). We instead use a
**two-phase lazy migration**:

### Phase 1 — Schema creation (runs in upgrade handler, instant)

The `version(17).stores(...)` call creates the empty `chatMessages` table. No
data is moved. The upgrade completes immediately.

### Phase 2 — Background data migration (runs via `chrome.alarms`)

A new `MessagesMigrationService` runs as a background alarm task, similar to
`JanitorService`:

```
chrome.alarms: "messages-migration" — fires every 30 s while migration is incomplete
```

Each alarm tick:
1. Reads progress cursor from `cache["messages-migration:cursor"]`
2. Loads the next **batch of 20 chats** (by `id > cursor`) that still have
   non-empty `messages` dicts in the `chats` table
3. For each chat:
   a. Writes its `messages` entries to `chatMessages` table (bulk put)
   b. Updates the `chats` row to `messages: {}` (modify)
4. Persists the new cursor to cache
5. Logs progress: `"Migrated N/2451 chats (batch took Xms)"`
6. When cursor exhausts all chats: writes `cache["messages-migration:complete"]`
   and cancels the alarm

**Batch size rationale**: 20 chats × average 10 messages × average 5 KB/message
= ~1 MB per tick. Well within SW memory budget; each tick completes in < 2 s.

### Fallback during migration

Until a chat's messages have been migrated, `chatMessages` has no rows for it.
All repository reads check both tables using the pattern:

```typescript
// chatRepository.ts
async function getChatMessages(chatId: string): Promise<Record<string, ChatMessage>> {
  const rows = await getDB().chatMessages.where("chatId").equals(chatId).toArray();

  if (rows.length > 0) {
    // Migrated path: messages are in their own table
    return Object.fromEntries(rows.map(m => [m.id, m]));
  }

  // Legacy path: messages still in the chats row (pre-migration)
  const chat = await getDB().chats.get(chatId);
  return chat?.messages ?? {};
}
```

This means the app works correctly **before, during, and after** migration, with
no user-visible change and no flag day.

## 7. New Repository API

All existing `getChat()` callers need to be audited. The new API makes the
cost of loading messages explicit:

```typescript
// chatRepository.ts — new and modified exports

/**
 * Get chat metadata only (fast — no messages loaded).
 * Equivalent to current getChat() for callers that only need title, source, etc.
 */
export async function getChat(chatId: string): Promise<Chat | undefined>
// UNCHANGED signature; messages field is {} in the returned object.

/**
 * Get a chat with its full messages dict assembled (slower — loads all messages).
 * Use only when you need to render or export the full conversation.
 */
export async function getChatWithMessages(chatId: string): Promise<Chat | undefined>

/**
 * Get only the messages on the primary branch (currentId → root via parentId).
 * O(depth) DB lookups via parentId index. Typical depth: 10–200 messages.
 * This is the efficient path for the CLI `get` and the Viewer default view.
 */
export async function getPrimaryBranchMessages(
  chatId: string
): Promise<ChatMessage[]>

/**
 * Get the messages on the branch leading to a specific leaf message.
 */
export async function getBranchMessages(
  chatId: string,
  leafMessageId: string
): Promise<ChatMessage[]>

/**
 * Get a paginated window of messages for a chat (for future viewer pagination).
 * Returns messages in chronological order on the primary branch.
 */
export async function getChatMessagePage(
  chatId: string,
  offset: number,
  limit: number
): Promise<{ messages: ChatMessage[]; total: number }>

/**
 * Get message count without loading any content.
 */
export async function getChatMessageCount(chatId: string): Promise<number>

/**
 * Save messages for a chat (called by saveChat during sync).
 * Replaces the inline messages dict write.
 */
export async function saveChatMessages(
  chatId: string,
  messages: Record<string, ChatMessage>
): Promise<void>

/**
 * Delete all messages for a chat (called by hardDeleteChats).
 */
export async function deleteChatMessages(chatId: string): Promise<void>
```

### Branch traversal for `getPrimaryBranchMessages`

The branch walk follows `parentId` chains. With messages in a separate table,
the efficient pattern is:

```typescript
async function getPrimaryBranchMessages(chatId: string): Promise<ChatMessage[]> {
  const chat = await getDB().chats.get(chatId);
  if (!chat?.currentId) return [];

  // Walk parentId chain from currentId to root, collecting IDs
  const branchIds: string[] = [];
  let currentId: string | undefined = chat.currentId;

  while (currentId) {
    branchIds.unshift(currentId);
    const msg = await getDB().chatMessages
      .where("[chatId+id]").equals([chatId, currentId])
      .first();
    currentId = msg?.parentId ?? undefined;
  }

  // Bulk-fetch all branch messages in one query
  const msgs = await getDB().chatMessages
    .where("[chatId+id]")
    .anyOf(branchIds.map(id => [chatId, id]))
    .toArray();

  // Re-sort to branch order (anyOf returns unordered)
  const byId = new Map(msgs.map(m => [m.id, m]));
  return branchIds.map(id => byId.get(id)).filter(Boolean) as ChatMessage[];
}
```

This is O(depth) individual reads for the walk + 1 bulk read for content —
much better than loading the entire messages dict.

## 9. CLI / wsBridge Impact (the immediate motivation)

After this migration, `wsBridge.handleGet` becomes:

```typescript
async function handleGet(id, req) {
  const chat = await SyncService.getChat(req.chatId);           // fast — no messages
  if (!chat) return { id, success: false, error: "Not found", code: "NOT_FOUND" };

  const messages = await SyncService.getPrimaryBranchMessages(req.chatId); // branch only
  return { id, success: true, data: { ...chat, messages: assembleDict(messages) } };
}
```

For a 240-message chat on a branch of depth 80:
- **Before**: load 240 messages (~15 MB), send 240 messages over WS → SW crash
- **After**: load 80 messages (~5 MB), send 80 messages over WS → no crash

The WS frame shrinks by the ratio `branch_depth / total_messages`, typically 2–5×
for branching conversations, and up to 10× for image-gen chats with hundreds of
stub branches.

## 10. Risks & Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Migration interrupted (browser closed mid-batch) | Medium | Cursor persisted after every batch; resumes cleanly on next alarm tick |
| Batch write fails for a specific chat | Low | Catch per-chat, log, skip, continue — failed chats retain inline messages and fall back gracefully |
| `parentId` chain walk hits orphaned message | Low | `while` loop terminates on `undefined` parentId; orphaned subtrees are simply not included in the branch |
| `chatMessages` table missing on DB open race | Very low | Dexie schema migration is atomic; table exists before any code runs |
| Performance regression for small chats | None | Small chats: branch walk is 5–20 steps + 1 bulk read; net faster than loading the full row |
| Export format regression | None | `getChatWithMessages` assembles identical dict; exporter is unchanged |

## 11. Implementation Phases

### Phase 1 — Schema + migration infrastructure (no behaviour change)
- `db.ts`: add v17 schema, `StoredChatMessage` type, `chatMessages` table declaration
- `messagesMigrationService.ts`: batch migrator with cursor + progress logging
- `init.ts`: register alarm, hook into alarm listener
- `chatRepository.ts`: add `getChatMessages` with legacy fallback (§6), add `saveChatMessages`, `deleteChatMessages`

### Phase 2 — Write path (new syncs write to both tables)
- `chatPersistence.ts`: `saveChat` writes `messages: {}` to `chats` row and all messages to `chatMessages`
- `hardDeleteChats`: cascade to `chatMessages`
- This ensures chats synced after Phase 1 ships are immediately in the new format

### Phase 3 — Read path migration (callers switch to new API)
- `wsBridge.ts`: `handleGet` → `getPrimaryBranchMessages`
- `statsWorker.ts`: `getChatMessageCount`
- `searchWorker.ts`: iterate `chatMessages`
- `bulkExportWorker.ts`: `getChatWithMessages`
- `viewer`: `getPrimaryBranchMessages` for initial load

### Phase 4 — Legacy cleanup (after migration is complete for all users)
- Remove legacy fallback from `getChatMessages` (the `chat.messages` path)
- Remove inline `messages` dict from the `chats` row storage entirely
- Can be a future version bump once migration telemetry confirms completion

## 12. Open Questions

1. **Viewer pagination**: Phase 3 gives the Viewer the primary branch efficiently.
   Should the Viewer lazily load other branches on demand (e.g., clicking a branch
   indicator)? This is a UX decision separate from the schema change.

2. **`getChatWithMessages` memory ceiling**: For the export worker and full-chat
   operations, we still load all messages. For extremely large chats (500+ msgs),
   even the assembled dict could be large. A future streaming export path could
   avoid ever assembling the full dict in memory — but that is out of scope here.

3. **`deleted` flag on `chatMessages`**: Should messages inherit the soft-delete
   state of their parent chat? Currently `hardDeleteChats` cascades the delete
   immediately, so soft-deleted chats' messages are cleaned up in the Janitor's
   hard-delete pass. No change needed — but worth confirming.

4. **Search index rebuild after migration**: Once messages are in the new table,
   the search worker needs to read from it. Should we force a Lunr rebuild when
   migration completes? Likely yes — add a `cache["search:rebuild-needed"]` flag
   that `init.ts` checks on startup after migration.
